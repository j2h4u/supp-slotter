<objective>
Refactor the relations ontology so the planner can express *how alarming* a relation is, and stop sitting on dead antagonizes data.

Three things ship together:
1. **Severity field** on the Relation dataclass + YAML schema + loader + warning emission — uniform across all four relation types (balance/supports/competes/antagonizes), optional, with a closed enum (critical / high / medium / low).
2. **Antagonizes wiring** — currently `antagonizes` rows are loaded and round-tripped to substance-card review output but never generate a planner warning. Wire a `collect_antagonizing_relations` that fires when *both* endpoints are active in the stack (different semantic than `balance`/`supports`, which fire when an endpoint is missing).
3. **D3 ↔ Magnesium supports edge** — encode `source: Magnesium, target: Vitamin D3, severity: critical` because Mg-dependent hydroxylase is required to activate D3. Follows the existing `supports` convention (source = supporter, target = supported; warning fires when supporter absent).

Plus ontology-facts.md is brought in sync with what the model can now express.

Purpose: severity gives the operator a triage signal in `schedule.yaml` review output; the antagonizes wire closes a real gap (today's stack contains Vit E + Vit K2 — that pair should be visibly flagged with severity, not silently loaded); D3 ↔ Mg is the canonical "critical" pairing that motivated the severity work in the first place.

Output: updated dataclass, loader, schema, YAML data, two warning emitters, ontology doc, one new test.
</objective>

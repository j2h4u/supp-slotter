**B1 — lift `declares_against` to module level as `_declares_against`:**

Currently `declares_against` is an inner function of `must_separate` with no closure captures
(it only uses its own params `traits_a`, `traits_b` and the outer param `trait_defs`).
`trait_defs` IS a closure capture, so pass it as a third explicit parameter:

```python
def _declares_against(
    traits_a: set[str], traits_b: set[str], trait_defs: dict[str, TraitDef]
) -> bool:
    for trait_id in traits_a:
        trait = trait_defs.get(trait_id)
        if trait is None:
            continue
        for sep in trait.separate_from:
            if sep in traits_b:
                return True
    return False
```

Place this definition before `must_separate` at module level. Rewrite `must_separate` to call it:

```python
def must_separate(
    t1: set[str], t2: set[str], trait_defs: dict[str, TraitDef]
) -> bool:
    """Symmetric: t1 and t2 share a slot conflict if either declares separate_from
    referencing a trait in the other."""
    return _declares_against(t1, t2, trait_defs) or _declares_against(t2, t1, trait_defs)
```

Delete the inner `def declares_against` and the `return declares_against(t1, t2) or declares_against(t2, t1)` line.

For plan.py: `cmd_plan` calls `cmd_check()` and on non-zero prints:
```

```
This wording conflates a lock-blocked skip with a real check failure. Add a parallel message
specifically for the lock case. The simplest approach: after `cmd_check()` returns non-zero,
emit:
```python
print("plan: skipped (maintenance lock held)", file=sys.stderr)
```
before the existing "plan aborted" print. Both messages can coexist — the operator sees the
check-level message ("check: skipped") and the plan-level message ("plan: skipped") together,
which is correct since they are distinct layers. The "plan aborted: check failed" line can
remain or be moved inside an `else` branch — do whichever is cleanest; the key requirement is
that "plan: skipped (maintenance lock held)" appears when `check_result != 0` due to a lock.

A clean implementation: replace the current unconditional "plan aborted" print with two cases:
```python
if check_result != 0:
    print("plan: skipped (maintenance lock held)", file=sys.stderr)
    return check_result
```
(Because when `cmd_check` returns non-zero solely due to a lock, `cmd_check` has already
printed the check-skipped message and the "plan aborted" message would be misleading. If
`cmd_check` fails for actual validation errors, "plan aborted: check failed" is emitted by the
code after the check is done—but that message appears in `cmd_check`'s own output via `report()`.)

Actually the simplest correct change with minimal disruption: just add the single print before
return:
```python
check_result = cmd_check()
if check_result != 0:
    print("plan: skipped (maintenance lock held)", file=sys.stderr)
    return check_result
```
and remove the old "plan aborted: check failed; fix errors above and retry." print.

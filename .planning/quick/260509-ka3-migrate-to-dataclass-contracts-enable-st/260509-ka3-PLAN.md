<objective>
Migrate every stable-shape yaml dict in planner/ to a frozen dataclass, replace
`(data, err)`-tuple loaders with raising loaders, eliminate the planner/cards/
re-export hub (back-compat scaffolding from the prior lift-and-shift split),
and switch pyright from `basic` + 5 rule downgrades to `strict` + zero
downgrades.

Today: pyright runs in `basic` mode with 78 warnings hidden behind 5 rule
downgrades (`reportArgumentType`, `reportIndexIssue`, `reportOperatorIssue`,
`reportOptionalMemberAccess`, `reportOptionalSubscript` all set to "warning").
The downgrades exist because every yaml load returns a `dict | None` that
nobody narrows, then code indexes it. The fix is structural: make the loader
return a typed dataclass, and the index/operator/optional warnings stop
existing — they cannot exist on a dataclass attribute.

Purpose: Make the type system actually load-bearing. With strict pyright +
dataclass-fronted yaml, every change to a card schema surfaces as a pyright
error at the touched call site, instead of silently shipping with a runtime
`AttributeError` waiting to happen. The cards/ package stops behaving like a
shrink-wrapped library (re-exports, deprecation cycles, back-compat
scaffolding) and behaves like what it actually is: a private subpackage with
one consumer.

Output: `uv run pyright` exits clean in strict mode. `uv run pytest` still
passes 48/48. `uv run python -m planner plan` regenerates schedule.yaml
byte-identical. The cards/__init__.py re-export hub is gone. No suppression
comments, no rule downgrades, no `# back-compat` framing anywhere.
</objective>

<gotchas>
1. **Schedule yaml byte-identity is non-negotiable.** `dump_schedule_yaml` uses
   `yaml.safe_dump(schedule, sort_keys=False, default_flow_style=False,
   allow_unicode=True)` which emits keys in dict-insertion order. Inside
   `cmd_plan`, `schedule` is built via dict mutation in a specific sequence
   (top-level keys, then per-pillbox, then per-slot, then warnings,
   action_points/review_contexts/placement_notes/summary appended last).
   Keep it as `dict[str, Any]` during assembly. Do NOT route schedule through
   `dataclasses.asdict()` — that would re-order keys to declaration order and
   break byte-identity. Only the inputs (Substance/Product/Slot/etc.) become
   dataclasses; the schedule output stays a plain dict.

2. **Maintenance writes back to data/ files.** `rewrite_substance_refs`,
   `rewrite_stack_product_refs`, `normalize_substances` etc. all do
   `path.write_text(yaml.safe_dump(<original_dict>, sort_keys=False, ...))`.
   Going through Substance dataclass + back to dict via `asdict` would
   re-order keys and silently churn every file (and tests like
   `test_check_auto_renames_files_when_names_change` round-trip via
   yaml.safe_dump and would still pass byte-noise — but unnecessary churn is
   off-policy). KEEP the maintenance round-trip on raw `load_yaml_mapping`
   dicts. Use the typed dataclass loaders ONLY for read-only consumption.
   Document this split at the top of maintenance.py with one short comment.

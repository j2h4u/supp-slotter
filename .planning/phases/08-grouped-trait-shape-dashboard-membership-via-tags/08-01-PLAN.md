# Plan 08-01: Stage 1 — Core Schema + Data + Contracts (Atomic Commit)

## Context

Stage 1 covers DT-01 through DT-10. All 10 tasks are correctness changes that must land in a **single atomic git commit** (per ROADMAP.md Phase 8 — "atomic single commit so `check` is never broken between commits").

**The atomic-commit decision is honored intentionally** (per Kaizen migration specialist rationale in ROADMAP.md). Schema, loaders, YAML data, checks, and generated output are mutually dependent — splitting Stage 1 leaves intermediate states where `planner check` fails. The compensating control is a **structured pre-commit verification gate** (task 08-01-14) that runs check + pytest + plan before any commit, plus a documented rollback runbook (task 08-01-15) that lets the operator revert with a single command if the gate fails.

### from_traits Resolution Semantics (canonical — used everywhere)

A dashboard's `from_traits` is an object keyed by namespace, each holding an array of bare slugs.

**Resolution rule (union / logical OR across the entire `from_traits` object):**

> A substance is a member of dashboard D if there exists at least one (namespace N, slug S) pair such that:
> - N appears as a key in D.from_traits
> - S appears in D.from_traits[N]
> - S appears in the substance's per-namespace field for N
>
> In set algebra: `members(D) = ⋃ over (N,S) in flatten(D.from_traits): { sub | S ∈ sub.<N> }`.
>
> There is NO `AND` semantic across namespace groups. Mixing namespaces in one `from_traits` widens the cluster (more members), never narrows it.

This rule MUST be implemented identically in `build_dashboard_review()` (cards/dashboards.py), in doctor lifecycle warnings (DT-14 — Stage 2), and described identically in docs (DT-11) and SKILL.md (DT-12). The Stage 1 plan implements it in `build_dashboard_review()` and adds a code-level docstring stating the rule verbatim so future readers (human or agent) see the canonical statement at the implementation site.

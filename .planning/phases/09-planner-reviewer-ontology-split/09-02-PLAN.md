<objective>
Introduce the v2 nested `schedule:` / `knowledge:` substance shape alongside
the existing v1 flat namespaces via a TRANSITIONAL schema (per-review fix:
the schema must accept both shapes during the migration window — see
09-REVIEWS.md HIGH finding 1). Update the `Substance` dataclass to mirror the
split, make `load_substance` a dual-format reader, and update every
downstream consumer that reads per-namespace substance fields.

Purpose: This plan completes the Planner/Reviewer separation at the code
boundary. The Planner-relevant fields (`intake`, `timing`, `activity`,
`prefer_with`) become structurally separate from the Reviewer-relevant fields
(`is_`, `effect`, `risk`, `dashboard`, `pathway`). The scheduler's trait
aggregation drops `risk:` and `effect:` (the three scheduling effect slugs
now arrive via `timing:`). The schema stays transitional through plan 03's
data migration; plan 05 tightens it to v2-only after removing the v1 loader
fallback.

Output: Transitional schema, contracts, dual-format loader, scheduler trait
aggregator, audit cleanup-section iterator, dashboards resolver, and
`cmd_review_substance` display loop — all aligned to the v2 Substance shape
while keeping v1 cards loadable until plan 03 migrates them.
</objective>

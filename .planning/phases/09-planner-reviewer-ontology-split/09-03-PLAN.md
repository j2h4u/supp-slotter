v2 substance card top-level keys (subset of):
  id, name, form, aliases, notes, concerns,

  v1.intake[]    â†’ v2.schedule.intake[]      (1 element max, preserved)
  v1.activity[]  â†’ v2.schedule.activity[]    (1 element max, preserved)
  v1.effect[]    â†’ split:
    if slug in TIMING_SLUGS â†’ v2.schedule.timing[]    (1 element max â€” RESEARCH confirms no card has more than 1 timing slug today)
    otherwise              â†’ v2.knowledge.effect[]
  v1.prefer_with[] â†’ v2.schedule.prefer_with[]
  v1.is[]        â†’ v2.knowledge.is[]
  v1.risk[]      â†’ v2.knowledge.risk[]
  v1.dashboard[] â†’ v2.knowledge.dashboard[]
  (v1 has no pathway field â†’ v2.knowledge.pathway is absent post-migration)

`default_flow_style=False`. Preserve original top-level key ordering for the

Dashboard YAML namespace keys: existing files (e.g.
data/dashboards/antioxidant_protection.yaml) use `from_traits: { is: [antioxidant] }`.
The `is` namespace key string maps to `Substance.is_` via the
`field_name = "is_" if namespace == "is" else namespace` rule in

leave the key as `is` and let substance_carries handle the mapping. We choose
that path.

Mixed-card rejection policy (per-09-REVIEWS.md MEDIUM finding 3): the
previous "if `"schedule" in data` or `"knowledge" in data` â†’ return
unchanged" rule is too permissive. A card with `schedule:` plus a stale
`intake:` at top level is a partial migration that the script should
SURFACE, not skip. The new rule:
  - If neither `schedule:` nor `knowledge:` is present â†’ migrate (v1 â†’ v2).
  - If `schedule:` and/or `knowledge:` is present AND no flat namespace keys
    are present at top level â†’ return unchanged (idempotent re-run).
  - If `schedule:` and/or `knowledge:` is present AND any flat namespace key
    is also present â†’ raise `ValueError(f"partial migration in {data.get('id','?')}: card has nested section AND flat keys {sorted(...)} â€” fix by hand")`.

Rewrite `schema/product.schema.json` so products are concrete formulas with required `id`, `name`, and `components`. `components` must be a non-empty array of objects with required `substance`; optional `label`, `amount`, and `notes`; `substance` uses the same id pattern as card filenames. Rewrite `schema/inventory.schema.json` so every `supplements.*` entry requires `product` and `stack`; allowed `stack` values stay `daily`, `training`, `inactive`; optional fields remain `dose`, `brand`, `notes`, and `traits_override`. `traits_override.add/remove` keep unique trait-id arrays.

Rewrite `schema/slots.schema.json` to allow only `label`, `order`, `stack`, `near`, and `food` on slots. `near` enum must be exactly `wake`, `breakfast`, `day_meal`, `sleep`, `workout_before`, `workout_after`; `food` is boolean; remove slot fields `time` and `activity`.

Extend `schema/traits.schema.json` only if needed so the existing trait structure can reference `near` and `food` match fields. Do not add MCP/import/database fields anywhere.

Retain `prefer_with` in Phase 2. It lives only on substances and references substance ids only. It must not reference product ids or inventory item ids in this phase. The current `creatine -> l_citrulline_malate` relationship is migrated from `data/products/creatine.yaml` to `data/substances/creatine.yaml` as:

```yaml
prefer_with:
  - l_citrulline_malate
```

No product formula may contain `prefer_with`.

Addresses review concern: trait-effect migration must be concrete. Convert old slot matches as follows:

- `time: morning` that meant empty morning becomes `near: wake`; if the old trait meant a breakfast-associated morning dose, use `near: breakfast`.
- `time: day` becomes `near: day_meal`.
- `time: evening` becomes `near: sleep`.
- `activity: pre_workout` becomes `near: workout_before`.
- `activity: post_workout` becomes `near: workout_after`.
- `activity: any_workout` becomes two equivalent preference effects, one for `near: workout_before` and one for `near: workout_after`.

After this migration, the `activity` namespace may remain as marker traits only if it is still useful for human-readable intent, but no scheduling effect may match `activity`.

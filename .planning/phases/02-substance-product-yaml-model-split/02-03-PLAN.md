Update `schedule.yaml` generation so each slot contains inventory item ids. Add explanation entries keyed by inventory item id with at least:

```yaml
product: <product_id>
components:
  - <substance_id>
slot: <slot_id>
slot_score: <int>
reasons:
  - "<trait_id> from <substance_id> match {...} -> prefer (+2)"
```

Warnings must also be keyed to inventory item/product context and include the source substance when a warning trait came from a component:

```yaml
- item: nattokinase
  product: nattokinase
  substance: nattokinase
  trait: risk:fibrinolytic_bleeding
  message: ...
```

If a single product has component traits that would create a same-slot `separate_from` conflict inside the product, do not block scheduling and do not split the components. Emit a warning with `type: intra_product_trait_conflict`.

Addresses review concern 02-03 explicitly:

1. Build each candidate's aggregated trait set from product components plus inventory overrides.
2. Before normal slot co-location checks, inspect that candidate's own aggregated traits for `separate_from` relationships where both sides come from the same product candidate.
3. Record those internal relationships as `intra_product_trait_conflict` warnings with item id, product id, both trait ids, and the source substance ids.
4. Exclude those internal relationships from the normal blocking/co-location loop. A candidate must never be rejected from a slot because it conflicts with itself.
5. Keep normal `separate_from` blocking for relationships between different scheduled inventory item ids/products. For example, if product A's aggregated traits conflict with product B's aggregated traits, normal co-location blocking still applies.
  </action>

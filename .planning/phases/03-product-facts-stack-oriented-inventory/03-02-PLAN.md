Rewrite `schema/inventory.schema.json` so the root object requires `version` and `stacks`.

`stacks` must require exactly:
- `daily`
- `training`
- `inactive`

Each stack is an object keyed by inventory item id. Each item must require `product` and may keep only inventory-owned fields:
- `product`
- `notes`
- `traits_override`

Do not allow per-item `stack`, `brand`, or `dose`.

Allow empty stack objects so `daily`, `training`, or `inactive` can be present with no entries.

Rewrite `data/inventory.yaml` into:

```yaml
version: 1
stacks:
  daily:
    item_id:
      product: product_id
  training:
    item_id:
      product: product_id
  inactive:
    item_id:
      product: product_id
```

Keep existing inventory item ids and product refs. Keep inventory-specific `traits_override` and operator-specific notes. Product facts were copied in Plan 03-01; do not preserve `brand` or `dose` in inventory.

Add a small domain-specific inventory loader/normalizer in `planner.py` that reads stack-oriented inventory and returns a mapping usable by existing planner logic, with each normalized entry carrying its stack in memory.

Use this normalizer in:
- `validate_inventory`
- `check_inventory_alignment`
- `check_inventory_overrides`
- `cmd_refresh`
- `cmd_plan`

Do not add a compatibility reader for the old `supplements` shape. This is a direct migration.

Ensure malformed inventory entries still report schema/validation errors and do not crash with `AttributeError`.
